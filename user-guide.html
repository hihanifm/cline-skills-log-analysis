Core Concepts

Two Things to Know

🔍

Template

A YAML file that defines what to look for in a log or PCAP file. Contains one or more filter patterns — regex for logs, tshark display filters for PCAPs. Lives in the shared library or your project repo. Reusable across any workflow.

Think of it as an ingredient — the specific pattern you want to extract.
📋

Workflow

A Markdown file with YAML frontmatter that defines a complete analysis. It wires together templates, points at input file types, and defines the final diagnostic question to ask the AI. It's what you actually "run".

Think of it as a recipe — it combines ingredients and describes the desired outcome.
Template Anatomy

What's in a Template

templates:
  - id: sipmsg referenced in workflow include:
    description: SIP REGISTER and response messages appears in report
    pattern: "SIPMSG" ripgrep regex
    context_lines: 3
    post_process: decode_sip.py optional
    summary_prompt: "Did registration succeed? Was there a 401 challenge?" LLM prompt per pattern

For PCAP templates: replace pattern with filter + fields for tshark field extraction.

Better Summary

Meaningful analysis

decode_sip.py

Turns dense SIPMSG log lines into a clean REGISTER → 401 → REGISTER → 200 OK flow. The LLM gets readable context, not raw timestamps.

your_decoder.py

Live in log-postprocessors/ in your project repo. Reads stdin, writes stdout. Reference by filename in the template's post_process: field.

Most templates don't need a post-processor. Use one when raw output would confuse the LLM.

Workflow Anatomy

What's in a Workflow

---
workflow: ims-registration name used for output folder
description: IMS registration failure analysis
default_max_lines: 200 cap per pattern
 
input:
  - path: "*.txt" file glob to match
    include:
      - log/sipmsg.yaml template from library
      - log/ril.yaml
 
final_summary_prompt: "Did IMS registration succeed? If not, what failed and why?" the big question
---

Running an Analysis

Three Steps, One Report

1

Open your project in VS Code with Cline

Make sure Cline is active in the sidebar. Your project should have the workflow file in .clinerules/workflows/.

2

Tell Cline what to analyze

Just describe it naturally. For example:
"Use the ims-registration workflow on /path/to/dumpstate.txt"
or: "Analyze this PCAP for IMS issues: capture.pcapng"

3

Get your report

Cline runs the pipeline and hands you two files:

out/<workflow>/report.md
out/<workflow>/summary.md
Extending the System

Creating & Updating

New Template

Add a filter pattern

log-template-creator
  • Describe what you're looking for
  • Cline asks for regex / tshark filter
  • Tests against a sample file
  • Writes to filters/log/ in your repo
New Workflow

Define an analysis

workflow-creator
  • Describe the problem you're diagnosing
  • Cline lists available templates
  • Pick templates or create new ones
  • Writes to .clinerules/workflows/
New Post-Processor

Transform raw output

postprocessors
  • Write a Python script: stdin → stdout
  • Drop in log-postprocessors/
  • Reference by filename in template
  • Commit alongside your templates

Commit filters/, log-postprocessors/ and .clinerules/workflows/ to share your work with the team.

Best Practices

What Actually Matters

Templates

✓

One concern per template. Don't mix wakelock and power in one file — keep them separate and composable.

✓

Write a clear description. It shows up in the report. If it's vague, the summary will be vague.

✓

Test against a real sample before committing. A pattern with zero matches on real data isn't useful.

✓

Use a post-processor when raw output is hard to read — raw SIP lines produce poor AI summaries.

Workflows

✓

Name after the problem, not the tool. ims-registration-failure not sip-log-analysis.

✓

Write the final prompt as a diagnostic question, not a data request. "Did registration succeed? Why not?" beats "Summarize the logs."

✓

Start with library templates — only add project-local ones when what you need doesn't exist.

✓

Commit workflows with your code. The person debugging the next failure will thank you.

Getting Started

Setup in Two Steps

1

Clone the repo and run setup

# Clone
git clone https://github.com/hihanifm/cline-skills-log-analysis.git
cd cline-skills-log-analysis

# Install skills (ripgrep + tshark must already be installed)
python3 setup.py --skip-cli
This deploys all skills to ~/.cline/skills/ and workflows to ~/Documents/Cline/Workflows/
2

Open your project in VS Code with Cline active

# Start Cline in VS Code, then ask it to run a workflow
"Use the ims-registration workflow on /path/to/dumpstate.txt"
That's it. No environment setup, no API keys needed for basic use.
